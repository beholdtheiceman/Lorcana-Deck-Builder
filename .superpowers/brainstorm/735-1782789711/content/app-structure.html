How far do we take the app structure refactor?

The UX audit calls for "real routes, not modals." Before we write a line of code, let's agree on scope. All three approaches keep the current violet-on-dark look and build the primitive component library.

A

Primitives + polish only

Build the design token system and core UI components (Button, Card, Badge, Skeleton, Toast, EmptyState). Apply them everywhere — Team Hub tabs, deck builder, playtest log. Keep the current modal/tab navigation as-is. No routing changes.

Pros

  • Lowest risk, no URL changes
  • Fastest to ship, immediate visible polish
  • No breaking changes to existing flows

Cons

  • Modal-in-modal stays; tabs not deep-linkable
  • Audit's "polished" definition requires real URLs
B

Primitives + route the Team Hub (Recommended)

Same primitive library as A, plus convert the Team Hub from modal-in-modal to real sub-routes: /team-hub/:id/roster, /team-hub/:id/playtest, etc. Hub tabs become a routed sub-nav. The deck builder stays as a single-page experience (it's already a route).

Pros

  • Kills the biggest UX pain point (nested modals)
  • Deep-linkable hub tabs — share a direct link to Roster, Pods, etc.
  • Back button works in the hub
  • Achieves the audit's "polished" definition

Cons

  • Moderate routing work (~1–2 days)
  • HubDetailModal needs to be restructured into a layout
C

Full redesign — persistent sidebar + all routes

Everything in B, plus replace the top nav with a persistent left sidebar (like naurcana), add a home command-center screen, route the deck builder with sub-pages, and split the 11K-line App.jsx into feature modules with route-level code-splitting.

Pros

  • Fully matches the benchmark; most "finished" feel
  • Code health dramatically improved

Cons

  • Largest scope — weeks of work
  • Sidebar is a layout change users will notice immediately
  • High risk of regressions in a 11K-line file
